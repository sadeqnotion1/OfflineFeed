How OfflineFeed fits together

OfflineFeed is a Windows desktop news/RSS aggregator (PySide6 + QML) with a TelegramDesktop-style 3-pane UI. A local Python gui_server.py HTTP backend (port 8080) aggregates feeds; the QML UI talks to it only over HTTP through a ChatBridge, so the backend stays untouched.

God nodes

The most-connected hubs — where structure and control flow concentrate.

  • gui_server.py (file, degree 36) — Local Feed Server: aggregates RSS/HTML/Telegram/Twitter sources, caches news, serves the API, reposts to Telegram.
  • ChatBridge (class, degree 23) — The imperative surface QML calls into (slots) and observes (properties/signals); owns settings, models and backend calls.
  • frontend/qml/Main.qml (file, degree 13) — Frameless root window wiring the 3-pane layout, reader overlay and toast to the bridge.
  • frontend/bridge.py (file) — Hosts the bridge + four list models + defensive HTTP helpers.
  • app.main() (function) — Creates the QApplication, instantiates models + ChatBridge, loads Main.qml.
  • frontend/debug.py (file) — The no-PySide6 diagnostics/logging hub imported by the launcher, doctor and app.

Surprising connections

  • UI button → Telegram — ChatView.onForwardChannelRequested reaches across the bridge and backend to repost a channel to an external Telegram destination.
  • ChatBridge → ui_settings.json — backend_port is re-read independently by app.py, bridge.py, doctor.py and run_offlinefeed.py — the same parsing logic is duplicated in four places instead of being centralized.
  • debug.py spans two runtimes — it deliberately imports no PySide6, letting the same module serve both the pre-Qt CLI/launcher and the GUI — an unusually load-bearing hub.
  • app.start_backend → gui_server — the GUI imports the backend and calls whichever entry point exists (start_server/main/run/start/serve) via getattr fallbacks — a deliberately loose, ambiguous coupling.

The "why" (rationale captured from the code)

  • Import QtSvg so icons render — loads the qsvg plugin; without it PySide6 renders SVG sources blank (why icons/logo were invisible).
  • Backend left UNCHANGED, talk over HTTP — the UI speaks only to the existing local HTTP API so aggregation/Telegram logic never breaks.
  • Defensive networking keeps the UI alive — every backend call retries once and degrades to an empty UI + toast instead of crashing.
  • Replace opaque "Code: 1" with real diagnostics — debug/doctor/launcher surface the actual traceback and a fix hint.
  • Disable QML disk cache — prevents cached legacy component conflicts on startup.
  • Suppress the backend's old web UI — start_backend patches webbrowser.open to a no-op so the legacy backend can't pop the old web UI.
  • Reader font separate from chat font — changing one never affects the other.
  • Bin keeps deleted posts recoverable — full article dicts survive feed refresh and stay readable.
